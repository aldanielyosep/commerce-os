## 1. Purpose

Mendefinisikan kontrak API Product SPU agar API dan admin-web sinkron tanpa ambiguity.

---

## 2. Global Contract Rules

- Semua endpoint wajib auth (JWT).
- Semua endpoint list wajib pagination, search, ordering.
- Error wajib menggunakan envelope konsisten.
- Product code auto-generated by system, immutable, non-overridable.
- Soft delete digunakan untuk archive semantics.

### 2.1 Cross-Reference

- Detail strategi implementasi domain product mengikuti RFC-120.
- Detail policy scope mengacu ke RFC-111.

## 3. Response Envelope Baseline

Success:

```json
{
  "success": true,
  "data": {},
  "meta": {}
}
```

Error:

```json
{
  "success": false,
  "message": "Validation failed",
  "errors": ["field is required"]
}
```

## 4. Product Endpoints

### 4.1 Product CRUD

- `GET /api/v1/products`
- `GET /api/v1/products/:id`
- `POST /api/v1/products`
- `PATCH /api/v1/products/:id`
- `DELETE /api/v1/products/:id` (archive/soft delete)
- `POST /api/v1/products/:id/restore`

List query params:

- `page` (default 1)
- `per_page` (default 20)
- `q` (product_code/product_name)
- `status`
- `order_by` (`created_at`, `product_name`, `product_code`, `status`)
- `order` (`asc`/`desc`)

### 4.2 Product Lifecycle

- `POST /api/v1/products/:id/activate`
- `POST /api/v1/products/:id/deactivate`

### 4.3 Product Images

- `GET /api/v1/products/:product_id/images`
- `POST /api/v1/products/:product_id/images`
- `PATCH /api/v1/products/:product_id/images/:id` (reorder, set cover, alt text)
- `DELETE /api/v1/products/:product_id/images/:id`

## 5. Payload Contract

### 5.1 Product Create Payload

```json
{
  "product": {
    "product_name": "Goodie Bag Dino",
    "department_id": 10,
    "category_id": 22,
    "sub_category_id": 37,
    "product_type_id": 5,
    "short_description": "Goodie bag premium",
    "description_richtext": {
      "type": "doc",
      "content": []
    },
    "status": "draft"
  }
}
```

Notes:

- `product_code` tidak diterima dari client.
- `product_code` dihasilkan otomatis di backend saat create.

### 5.2 Product Update Payload

```json
{
  "product": {
    "product_name": "Goodie Bag Dino New",
    "short_description": "Update description",
    "description_richtext": {
      "type": "doc",
      "content": []
    }
  }
}
```

Validation:

- Payload update yang menyertakan `product_code` wajib ditolak (`422`).

### 5.3 Image Upload Payload

Multipart form data:

- `image` (required)
- `alt_text` (optional)
- `is_cover` (optional boolean)
- `position` (optional integer)

## 6. Validation Rules

### 6.1 Product

- `product_name` wajib.
- `product_code` unik per company.
- `product_code` immutable setelah create.
- `status` harus enum valid (`draft`, `active`, `inactive`, `archived`).

### 6.2 Description

- Rich text object harus valid schema minimal editor yang dipakai.
- Server menurunkan plain-text version untuk indexing/search.

### 6.3 Images

- Allowed extension: `jpg`, `jpeg`, `png`, `webp`.
- Max file size: 5 MB per file.
- Main ratio target: 1:1.
- Recommended dimension: 1200x1200 px.
- Minimum dimension: 1000x1000 px.

---

## 7. Validation Error Examples

Invalid extension:

```json
{
  "success": false,
  "message": "Validation failed",
  "errors": ["image extension is not allowed (jpg, jpeg, png, webp)"]
}
```

File too large:

```json
{
  "success": false,
  "message": "Validation failed",
  "errors": ["image size exceeds 5 MB"]
}
```

Product code override attempt:

```json
{
  "success": false,
  "message": "Validation failed",
  "errors": ["product_code is system-generated and cannot be changed"]
}
```

---

## 8. Authorization Contract

- Semua endpoint wajib policy-based authorization.
- Scope akses dibatasi ke company yang diizinkan.
- Akses out-of-scope wajib `403`.

---

## 9. Error Semantics

- `401 Unauthorized`: token tidak valid/tidak ada.
- `403 Forbidden`: tidak punya scope akses.
- `404 Not Found`: resource tidak ditemukan atau di luar scope.
- `422 Unprocessable Entity`: validasi payload gagal.
